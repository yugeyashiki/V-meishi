## 1. プロジェクト概要

### アプリ名
**V名刺**（ブイメイシ）

### コンセプト
VtuberやMMDクリエイターが、自分の3Dアバター（MMDモデル）を使ったインタラクティブな名刺をWebブラウザ上で作成・共有できるアプリケーション。

### ターゲットユーザー
- Vtuber（個人・企業問わず）
- MMDクリエイター
- 3Dアバターを所持している配信者・コンテンツクリエイター

### 主な特徴
- MMDモデル（.pmx）を使った3D名刺
- マウス/タッチ操作でアバターの角度をリアルタイムに変更可能
- バックエンド・DB不要の静的Webアプリ
- QRコードによるスマホでの共有・表示に対応

### フェーズ3（優先順位：A＞C＞B＞D）

**A. セキュアなWebサービス化（最優先・設計確定）**

> サーバーインフラ: **Supabase**（DB / Storage / Edge Functions）
> 閲覧方式: **QRコードを持っている人だけ閲覧可能**（UUID付きURL）

*アップロード時のフロー*
```
PMX読み込み
→ Three.jsジオメトリに変換（PMX構造を失わせる）
→ AES-GCMで暗号化（Web Crypto API）
→ 暗号化バイナリ → Supabase Storage
→ プロフィール・テーマ・暗号化キー → Supabase DB
→ UUID発行 → QRコードに埋め込む
```

*閲覧時のフロー*
```
QRコードを読む
→ https://[ドメイン]/card/{UUID} にアクセス
→ SupabaseからUUIDに紐づくデータ・キーを取得
→ Storageから暗号化バイナリを取得
→ メモリ上でAES-GCM復号（ディスクに書かない）
→ Three.jsに直接渡して描画
```

*DBスキーマ（Supabase PostgreSQL）*
```sql
create table cards (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz default now(),
  expires_at timestamptz,
  name text not null,
  catchphrase text,
  organization text,
  genre text,
  links jsonb,
  theme jsonb,
  pose jsonb,
  model_storage_path text,
  encryption_key text,
  motion_storage_path text,
  view_count integer default 0,
  app_version text
);
```

*実装ステップ*
```
Step 1: Supabaseプロジェクト作成・環境変数設定
Step 2: 独自フォーマット変換モジュール（src/core/encoder.js）
Step 3: AES-GCM暗号化/復号モジュール（src/core/crypto.js）
Step 4: Supabase連携モジュール（src/core/supabase.js）
Step 5: カスタムMMDローダー（復号→Three.js）
Step 6: 閲覧専用URLページ（/card/{UUID}）
Step 7: QRコードにUUID付きURLを埋め込む対応
Step 8: 動作確認・セキュリティ検証
```

*セキュリティ方針*
- 完全なリッピング防止は不可能という前提のもと「抑止コストを上げる」設計
- PMX構造はジオメトリ変換で失われるため元ファイルの復元は困難
- AES-GCMキーはSupabase DBで管理・クライアントに直接露出しない設計
- URLは推測困難なUUIDのみ（総当たり攻撃は現実的でない）
- 利用規約による法的保護と技術的抑止を組み合わせる

**C. 2Dアバター対応**
- 3Dモデルを持っていないユーザーへの間口を広げる
- Live2D Cubism SDK for Webの利用を検討
- 将来的には3D（MMD / VRM）/ 2Dをユーザーが選択できる統一UIを目指す

**B. VRMモデル対応**（.vrm）
- Vtuberに広く普及しているフォーマットへの対応
- @pixiv/three-vrmを使用（既存VRMプロジェクトを参照）
- モーション再生の課題を解決した上で実装

**D. PWA化・スマホアプリ化**
- PWA対応によるホーム画面への追加
- App Store / Google Play へのリリース検討
- ジャイロ操作への切り替え

## 4. 名刺デザイン仕様

### レイアウト

**基本レイアウト：縦型カード（B+C複合）**

```
┌──────────────┐
│              │
│    [3D]      │  ← アバターを上部大きく表示
│              │
│  名前         │
│  キャッチコピー│  ← テキスト情報を下部にまとめる
│  SNS ●●●   │
└──────────────┘
```

- スマホ縦画面を基本とした縦型カード
- PCでは横長（左アバター・右テキスト）に自動切り替え（レスポンシブ対応）
- アバターが主役となる構成

### カラーカスタマイズ

ユーザーが自由にカスタマイズできる項目：

| 項目 | 内容 |
|------|------|
| 背景タイプ | 単色 / グラデーション（2色指定） / 画像アップロード |
| 背景色 | カラーピッカー or プリセットから選択 |
| テキスト色 | 背景に合わせて自動提案 or 手動選択 |
| アクセントカラー | 名前下ライン・SNSアイコン背景などに適用 |

**デフォルト：ダーク単色**（Canvaデザイン確認済み）

**カラープリセット（初期提供）**

| プリセット名 | 背景タイプ | 背景色 | テキスト | アクセント |
|------------|----------|--------|----------|-----------|
| ダーク（デフォルト） | 単色 | #0D0D0D | #FFFFFF | #9B59B6 |
| ダークグラデーション | グラデーション | #1a1a2e → #16213e | #FFFFFF | #9B59B6 |
| ネオン | 単色 | #0A0A0A | #FFFFFF | #00FFFF |
| ライト | 単色 | #FFFFFF | #222222 | #FF6B9D |
| ナチュラル | 単色 | #F5F0E8 | #3D2B1F | #8B6F47 |

### デザイン詳細仕様

Canvaデザイン（パターン1）を基準とした実装仕様：

**レイアウト比率**

| エリア | 高さ比率 | 内容 |
|--------|---------|------|
| アバターエリア | 上部 約60% | 3Dモデル表示・背景透過 |
| テキストエリア | 下部 約40% | 名前・キャッチコピー・SNSリンク |

**タイポグラフィ**

| 要素 | フォントサイズ | ウェイト | 備考 |
|------|-------------|---------|------|
| 名前 | 28〜32px | Bold | 日本語対応フォント |
| キャッチコピー | 14〜16px | Regular | 名前の下に配置 |
| SNSリンクテキスト | 13〜14px | Regular | アイコンの右に配置 |

**アクセントライン**

| 項目 | 値 |
|------|----|
| 位置 | 名前テキストの直下 |
| 太さ | 2〜3px |
| 幅 | 名前テキスト幅に合わせる（または固定60px程度） |
| カラー | アクセントカラーに連動 |

**SNSアイコン**

| 項目 | 値 |
|------|----|
| アイコン形状 | 角丸の正方形背景（黒） |
| サイズ | 36〜40px |
| 並び | 縦並び・左寄せ |
| テキスト | アイコン右にURL/ID表示 |

## 5. 名刺データのJSONスキーマ

```json
{
  "version": "1.0",
  "profile": {
    "name": "string",              // 表示名（必須）
    "catchphrase": "string",       // キャッチコピー（任意）
    "genre": "string",             // 活動ジャンル（任意）例: "ゲーム / 雑談"
    "links": [
      {
        "platform": "string",      // "X" | "YouTube" | "Twitch" | "TikTok" | "other"
        "url": "string",
        "label": "string"          // 任意の表示名
      }
    ]
  },
  "avatar": {
    "modelFileName": "string",     // ローカルファイル名（参照用）
    "motionFileName": "string",    // ローカルファイル名（任意）
    "defaultPose": {
      "rotationX": 0,              // 初期角度（度数）
      "rotationY": 0,
      "zoom": 1.0
    },
    "background": {
      "type": "color" | "image",
      "value": "string"            // カラーコードまたは画像URL
    }
  },
  "meta": {
    "createdAt": "ISO8601",
    "appVersion": "string"
  }
}
```

> **注意**: モデルファイル（.pmx）本体はJSONに含めない。フェーズ1ではモデルはローカルから毎回読み込む設計とする。
> フェーズ2以降の共有方式は「エンコードデータ＋サーバー管理方式」を採用予定（詳細は未決定事項参照）。

## 9. 非機能要件

### レスポンシブ・スマホ対応
- スマホ（iOS Safari / Android Chrome）での動作を必須とする
- タッチ操作（スワイプによる角度変更）に対応
- フェーズ1からスマホ前提のレイアウトで実装する
- **✅ スマホ実機確認済み**（MMD描画・タッチ操作ともに問題なし）

### パフォーマンス
- 軽量モデル（ポリゴン数目安: 〜3万ポリ程度）での快適動作を最低基準とする
- 重いモデル使用時の警告表示を検討
- スマホGPUでの描画最適化（シェーダー簡略化など）を必要に応じて対応

### データ保持
- フェーズ1・2はサーバーサイド不要（静的Webアプリ）✅
- モデルファイルはローカル読み込みのみ（サーバーへのアップロードなし）
- 名刺コレクション（フェーズ2）はlocalStorageに保存 ✅
- フェーズ3でサーバー導入予定（エンコードデータの管理）

### セキュリティ
- フェーズ1・2: モデルファイル・プロフィール情報は外部に送信しない ✅
- フェーズ3-A: AES-GCM暗号化によるモデルデータ保護
- PMX構造をジオメトリ変換で失わせてリッピングコストを上げる
- 暗号化キーはSupabase DBで管理・クライアントに直接露出しない
- QRコードのUUID方式でアクセス制御（ログイン不要）
